# Business Discovery - 新規事業領域の発見とスクリーニング

「どの領域で戦うべきか」を構造的に探索し、候補を絞り込む。
既知の事業を評価する `/business-evaluation` とは異なり、**候補の発見と選定** に特化する。

## 位置付け

```
/business-discovery → 候補3-5領域を選定
        ↓
/business-evaluation   → 選定領域を投資メモ形式で深掘り評価
```

## Freedom Level: 中

推奨パイプラインに沿いつつ、業界・文脈に応じて軸の重みや候補数を調整してよい。

## 前提ルール（厳守）

1. **Fact / Assumption / Unknown を混ぜない** — 各主張に Evidence tag を付与
2. **「それっぽい分析」で止めない** — 足切り/選定の根拠を具体的に示す
3. **抽象度のギャップを許容しない** — 「食品製造」ではなく「惣菜盛付けラインの検品工程」まで解像度を上げる

## 7-Step Pipeline

### Step 0: 入力の確認

`$ARGUMENTS` を解析し、不足は AskUserQuestion で聞く。

必要な入力:
- **自社の概要**: 何をやっている会社か（事業・技術・アセット）
- **探索テーマ**: どんな方向性を考えているか（業界・技術・顧客セグメント）
- **制約**: やらないこと、使えるリソース、期限
- **ステークホルダーの関心**: 経営陣・投資家が重視するテーマ（あれば）
- **参考資料**: 既存の検討資料、URL等

### Step 1: 事業スタック分解 (Value Stack Decomposition)

探索テーマの業界を **5-8層のスタック** に分解し、各層の参入障壁・自社適性を評価する。

| 評価項目 | 問い |
|----------|------|
| 各層の内容 | 何が行われているか |
| 代表プレイヤー | 誰が支配しているか |
| 参入障壁 | 低/中/高/極めて高 |
| 自社適性 | ×/△/○/◎ + 理由 |

**出力**: 「自社が戦うべき層」の特定と根拠。
ハードウェアや基盤技術など参入障壁が極めて高い層は早期に除外する。

### Step 2: 失敗パターン・ガードレール (Failure Pattern Guard)

探索テーマの業界における **致命的失敗パターン** を5-10個調査・整理する。

情報源:
- 業界の失敗事例（Web検索）
- VC/アナリストの失敗分析レポート
- 類似スタートアップの撤退理由

各パターンから **設計原則（ガードレール）** を導出する。
例: 「HW死の谷」→「ハードは持たない」

**出力**: 失敗パターン表 + ガードレール一覧。
以降のステップでガードレール違反がないか常にチェックする。

### Step 3: 購買トリガー分析 (Buying Trigger Analysis)

顧客が金を払う理由を **5-8カテゴリ** に分類する。

汎用テンプレート（業界に応じてカスタマイズ）:

| トリガー | 概要 | 支払い動機の強さ |
|----------|------|----------------|
| A. 労働力不足 | 人が集まらない・辞める | ★★★★☆ |
| B. 品質 | 不良・ばらつき・リコール | ★★★★☆ |
| C. 生産性 | 効率・スループット | ★★★☆☆ |
| D. 安全・法規制 | Must-have。罰則・資格喪失 | ★★★★★ |
| E. 知識・データ | 暗黙知の消失・技術断絶 | ★★★☆☆ |
| F. レジリエンス | 事業継続・サプライチェーン | ★★★★☆ |
| G. サステナビリティ | ESG・環境規制 | ★★★☆☆ |

**重要な原則**: D（安全・法規制）+ F（レジリエンス）が強い領域は意思決定が速く価格感度が低い。C（生産性）のみの領域は価格競争に陥りやすい。

**出力**: 業界別の購買トリガーマップ。

### Step 4: Right-to-Win スコアカード設計

以下の **7軸** で候補領域を評価する（6軸 + Codexレビューで追加されたOperability）。

| 軸 | 問い | ◎の条件 | ×の条件 |
|----|------|---------|---------|
| 1. Desirability | 「ないと事業が回らない」レベルか？ | Must-have + 法規制 + 複数トリガー重複 | 課題認識薄い / 予算なし |
| 2. Feasibility | 今の技術で現場で安定動作するか？ | 構造化タスク + 制御環境 + 実証済み | 技術未成熟 / 安全要件極高 |
| 3. Operability | 導入後に現場で回し続けられるか？ | 停止許容あり + 保守容易 + 教育負荷低 | MTBF短い / 現地常駐必要 |
| 4. Viability | 受託SIにならずスケールできるか？ | テンプレ横展開 + RaaS + 粗利40%+ + 回収18ヶ月以内 | 毎回カスタム / ソフト価値薄い |
| 5. Defensibility | 持続的に勝てるポジションを築けるか？ | ワークフロー埋め込み + データFW + スイッチングコスト高 | コモディティ化必至 |
| 6. Executability | 今のリソースでPoC完了できるか？ | 初期顧客到達可 + PoC6ヶ月以内 + 業界知識あり | 規制障壁高 / 初期投資巨額 |
| 7. Strategic Fit | 「なぜ自社がやるか」に答えられるか？ | 既存能力活用 + 経営整合 + ピボット余地 | 既存事業と矛盾 |

**足切りルール（2種類に分離）**:

- **Structural No-Go**: 規制・資本・特許・市場集中による構造的障壁 → 即落とす
- **Fixable Gap**: 業界知識・チャネル不足 → 補完条件を明記して残す

### Step 5: ブロードスクリーニング (Broad Screening)

Step 1で特定した層に属する **8-15の候補セクター** を列挙し、マトリクス評価する。

**解像度の要求**: セクター名だけでなく、具体的な工程・サブセグメントまで記述する。
- Bad: 「食品製造」
- Good: 「食品製造 > 惣菜盛付(具材→盛付→蓋閉め→検品) / 水産加工(フィレ→骨取り→検品)」

**評価マトリクス**:

| セクター | 判定 | 根拠 | トリガーA | ... | トリガーG | 軸1 | ... | 軸7 |
|----------|------|------|-----------|-----|-----------|-----|-----|-----|

判定: 選定 / 再検討(条件付) / △多で落選 / 足切り(理由)

**出力**: マトリクス + 足切り理由の詳細説明 + 選定3-5領域。

### Step 6: ウェッジプロダクトカード (Wedge Product Card)

選定した各領域について、以下を具体化する。

```markdown
## [領域名] — プロダクトカード

### 現場のペイン（顧客の生の声を想定）
> 「○○が△△で毎日×時間取られる」

### 市場の特徴
- 市場規模、事業所数、人手不足指標
- 法規制の後押し

### Buyer Map（Codex追加）
| 役割 | 態度 | 決裁権 |
|------|------|--------|
| 現場責任者 | 推進者 | なし |
| 品質保証 | 中立→推進 | 技術承認 |
| 工場長/経営 | 決裁者 | あり |

### 初期プロダクト構成
- HW: ...
- SW: ...
- AI: ...
- 既存アセット連携: ...

### 定量効果
| 指標 | 導入前 | 導入後 | 改善 |

### 競合と価格帯
### Unit Economics（Codex追加）
- CAC / 粗利 / 導入費 / 回収期間 / 解約率推定

### 主要リスク
### 拡張パス（Wedge to Expansion）
初期プロダクトから隣接領域への拡張経路
```

**Evidence Quality**: 各主張に [Fact] [Assumption] [Unknown] を付与する。

### Step 7: 前提再検証ループ (Bayesian Update)

全ステップ完了後、以下を実施する:

1. **新情報の棚卸し**: 調査中に得た、初期前提を覆す情報はあるか？
2. **足切り再検証**: Structural No-Go としたものに、パートナーシップ等で突破可能なものはないか？
3. **ガードレール違反チェック**: Step 2のガードレールに抵触する選定はないか？
4. **Learning Agenda**: 低スコア項目に90日検証課題を設定

**出力**: 前提変更があれば Step 5 のマトリクスを更新。

```markdown

_生成日: {date}_

## Executive Summary
選定領域3-5つ + 推奨の優先順位 + 条件

## 2. 失敗パターン・ガードレール
## 3. 購買トリガーマップ
## 4. スコアカード定義
## 5. ブロードスクリーニング・マトリクス
## 6. ウェッジプロダクトカード（領域別）
## 7. 前提再検証・Learning Agenda

## Appendix: Evidence Quality 分離表

### Facts
| # | 主張 | 出典 | 検証状況 |

### Assumptions
| # | 仮説 | 間違った場合の影響 | 検証方法 |

### Unknowns
| # | 不明事項 | 必要なデータ |

## 推奨ネクストアクション
- [ ] 各領域のヒアリング先 1-2社を特定
- [ ] 最優先領域のMVP仕様概算
- [ ] /business-evaluation で深掘り評価
```

## 調査の進め方

- Step 1-3 は **並列エージェント** で同時調査可能
- Step 5 の各セクター調査も並列化を推奨
- Step 6 の各領域のプロダクトカードも並列化を推奨
- Web検索は業界レポート、VC分析、失敗事例、政府統計を優先
