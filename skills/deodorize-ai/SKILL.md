# Deodorize AI - AI臭を消す文章校正

AIが生成した文章から「AI臭」を検出・除去し、人間が書いたように読める自然な日本語に書き換える。

## When to Use

- AIが書いた文章を人間らしく直したいとき
- ブログ・note・SNS投稿の下書きを校正するとき
- AIっぽさが気になる文章をチェックしたいとき
- クライアント納品前にAI臭を消したいとき

## 実行手順

### Step 1: 入力の受け取り

`$ARGUMENTS` を解析する。

- テキストが直接渡された場合 → そのまま使う
- ファイルパスが渡された場合 → Read で読み込む
- 何も渡されなかった場合 → AskUserQuestion で校正対象を聞く

### Step 2: AI臭の診断

以下の「AI臭6パターン」に照らして、該当箇所を洗い出す。
各パターンの該当数を数え、診断レポートとして提示する。

**出力フォーマット:**

```
## AI臭 診断結果

| パターン | 該当数 | 深刻度 |
|----------|--------|--------|
| 記号の残骸 | N箇所 | 高/中/低 |
| リズムの単調さ | N箇所 | 高/中/低 |
| 説明書っぽい進行 | N箇所 | 高/中/低 |
| 事なかれ主義 | N箇所 | 高/中/低 |
| 抽象語・万能語 | N箇所 | 高/中/低 |
| 不自然な比喩 | N箇所 | 高/中/低 |

### 具体的な指摘
- [行番号or該当箇所]: 指摘内容 → 修正案
```

### Step 3: ユーザー確認

診断結果を提示し、以下を AskUserQuestion で確認する。

- 全修正を適用するか、一部だけか
- 文体の好み（ですます調 / だ・である調 / 口語調）
- 想定読者（ビジネス / カジュアル / 技術者向け）

### Step 4: 書き換え実行

診断結果と文体指定に基づいて全文を書き換える。
書き換えは「校正ルール」セクションに厳密に従う。

### Step 5: 差分提示

元の文章と書き換え後を並べて提示し、主な変更点をまとめる。

## AI臭 6パターン（検出基準）

### パターン1: 記号の残骸

Markdown記法や英語圏の記号習慣がそのまま残っている。

**検出対象:**
- `**太字**` が残っている（アスタリスク）
- ダッシュ「—」で言い換えをつなぐ
- 「」の多用、「」の中に『』がネスト
- ：の直後に半角スペース（「目的： 〜」）
- （）で補足しすぎる、（）内で責任回避
- ／（全角スラッシュ）で概念を並列

**なぜ起きる:** Markdown記法の消し忘れ、英語のコロン+スペース慣習の混入。

### パターン2: リズムの単調さ

文のテンポが均一で、読んでいて引っかかりがない。

**検出対象:**
- 同じ語尾が3回以上連続（〜です。〜です。〜です。）
- 接続詞の過剰使用（さらに、また、したがって、そのため、結果として）
- 文章の温度が一定（重要な箇所も軽い補足も同じテンション）
- 段落の終わりが毎回きれいに閉じる（以上がポイントです。次に進みましょう。）
- 「not A but B」構文の頻出（これは〜ではありません。〜です。）

**なぜ起きる:** 統計的に「次に来やすい言葉」を選び続けるため、平均的な型に収束する。

### パターン3: 説明書っぽい進行

構造を過剰に宣言し、マニュアルのような文体になる。

**検出対象:**
- 前置きが長い（ご質問ありがとうございます。〜は非常に重要なテーマです。以下では〜）
- 「結論から言うと」で始まるが結論が抽象的
- 構造宣言の繰り返し（以下の3つの観点から説明します。それぞれ順に見ていきましょう。）
- 「ステップ1：」「STEP」「Step」の連発
- ：と半角スペースのセット

**なぜ起きる:** 学習元にマニュアル・FAQ・研修資料が多い。

### パターン4: 事なかれ主義なスタンス

賛否・立場を明確にせず、全方位に配慮した文章になる。

**検出対象:**
- 保険表現（一概には言えませんが、一般的には有効だと考えられます）
- 過剰な中立（メリットもあればデメリットもあります。賛否が分かれるテーマです。）
- 否定が弱い（あまり推奨されません。注意が必要です。）
- 逃げ表現（ケースバイケース、場合によります、価値観によります）

**なぜ起きる:** 安全制御（有害発言回避）の副作用。言い切りを回避するよう調教されている。

### パターン5: 抽象語・万能語への偏り

具体的なイメージが湧かない、中身のない強い表現。

**検出対象:**
- 抽象語の空回り（本質を押さえ、最適化し、価値を最大化する）
- 根拠のない強い評価（非常に有効です。大きなメリットがあります。）
- 同義語の言い換え連打（重要・大切・欠かせない）
- 具体の絵が浮かばない文章

**なぜ起きる:** 具体例を持たないため、抽象語で埋め合わせる。

### パターン6: 不自然な比喩

AIが好む定番の比喩表現。

**検出対象（頻出ワード）:**
地図、羅針盤（コンパス）、設計図、土台、柱、エンジン、潤滑油、車の両輪、スパイス、レシピ、栄養、筋トレ（筋肉）、DNA、仕様書

**なぜ起きる:** 学習データ中の頻度が高く、「次に来やすい比喩」として選ばれる。

## 校正ルール（書き換え時の厳守事項）

### 内容に関するルール

- 内容の捏造、根拠のない具体化はしない。元にない数字・固有名詞・事例は足さない
- 曖昧な箇所は曖昧なまま。ただし文章として読みやすく整える
- 「結論から言うと」「本記事では」「以下で解説します」などの前置き宣言は削除。いきなり本文として書き出す
- 「一般的に」「多くの場合」「状況によって異なります」などの安全クッションは原則削除
- 抽象語だけで押し切らず、元文の範囲で「何がどうなるか」が伝わる動詞中心の表現に置き換え
- 同義語の言い換え連打はやめ、1回で言い切る
- 「まとめると」「要するに」の繰り返しや同内容の再掲は削る
- 文のリズムを均一にしない。短い文と長い文を混ぜ、同じ型の連続を避ける
- 接続詞は必要最小限。特に順接の接続詞は削ってよい
- 書き手の人称を混ぜない（私／筆者／私たちなど）

### 記号・表記に関するルール（最重要）

- Markdown記法を使わない（`**太字**`、`##`見出し、箇条書き記号、装飾は禁止）
- 「」を多用しない。強調のためのカギ括弧は削り、文脈に溶かす
- ()を多用しない。補足は本文に組み込む
- コロン「：」は原則使わない。使う場合も直後に半角スペースを入れない
- ／（全角スラッシュ）で概念を並列しない
- →や疑似コード風の表記も避け、文章として書く
- 締めの定型句（「参考になれば幸いです」「まずは小さく始めましょう」）は入れない

### 出力ルール

- 書き換え後の文章だけを出力する（解説・チェックリストは出さない）
- 元の段落構成は大きく崩さず、読みやすい段落に整える
- 文章量は元文から大きく増減させない（目安：±20%以内）

## 補足: 人間側の「崩し」テクニック

校正だけでは消しきれないAI臭がある場合、以下のアドバイスを添える。

### バランスを崩す
スタンスを取る。メリット/デメリットのどちらが大きいか、どの案がイチオシか、賛成か反対か。思い切って言い切ると、賛否を生む文章になる。

### 客観を崩す
主観と一次情報を盛り込む。偏った考え、尖った価値観を入れる。AIが客観性を担保してくれるからこそ、人間は主観に身を委ねてよい。

### 論理を崩す
結論に行きつくまでの葛藤や感情をあえて残す。話の粒度が揃っていなくても、雑談が混じっていても、それが人間らしさ。ビジネス文書はロジカルに整えるべきだが、読み物系は思考過程をさらけ出すほうがAI臭は消える。
